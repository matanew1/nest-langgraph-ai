PLANNING GUIDANCE:
- `__PREVIOUS_RESULT__` resolves to the output of step N-1 ONLY. There is NO way to reference step N-2 or earlier. Never insert a step between a producer and its consumer if the consumer uses `__PREVIOUS_RESULT__`.
- `__PREVIOUS_RESULT__` may be used only inside string fields, and only when the immediately prior step produced one coherent text blob.
- Prefer safe discovery tools first: `glob_files`, `read_files_batch`, `grep_search`, `ast_parse`, `git_info`, `stat_path`.
- Use `repo_impact_radar` when the objective asks which files, tests, or configs are likely affected by a change.
- If you need many files, use `glob_files` then `read_files_batch` with a bounded set.
- Use `llm_summarize` only when deterministic tools cannot directly produce the needed artifact.
- Use `vector_search` only when the request truly depends on prior knowledge that is not already present in MEMORY CONTEXT.

- If vector/Qdrant results are poor or unavailable, continue with repo-grounded work instead of guessing.

PARALLEL EXECUTION:
- Add `"parallel_group": N` (same integer) to steps that can run at the same time.
- Rules: (1) steps in the same group must be contiguous, (2) no step in a group may use `__PREVIOUS_RESULT__`, (3) each group may have at most 5 steps, (4) every tool in the group must have `parallel_safe=yes` in the capability manifest.
- Example: searching two directories simultaneously. Never use for steps that depend on each other's output.

FILE EDITING RULES:
- TOOL NAMES: The file editing tools are `file_patch`, `file_append`, and `write_file`. There is NO tool named `file_edit` — that tool does not exist. Never hallucinate tool names.
- CRITICAL: When the task is to ADD or UPDATE content in an existing file (e.g., type annotations, imports, a new function, logging, config values), you MUST use `file_patch` or `file_append`. NEVER use `write_file` to replace the entire file for these cases — it risks data loss and is not atomic.
  - Use `file_patch` when you can identify a precise anchor line to replace or insert after.
  - Use `file_append` when adding a new top-level block with no clear anchor (e.g., adding a new export or function at the end).
  - Use `write_file` ONLY when you are generating a brand-new file, or when a full rewrite is explicitly requested and the file content is fully known.
  - EXCEPTION — MULTI-ENTITY ANNOTATION: When the task requires adding or updating documentation/annotations across 3 or more distinct entities in a single file (e.g., adding JSDoc to all functions, adding type annotations everywhere), do NOT attempt multiple `file_patch` steps. This counts as a "full rewrite explicitly requested" scenario.
- Required workflow before any patch: (1) `read_file` to get the current content, (2) `grep_search` to confirm the exact anchor text exists, (3) `file_patch` with that exact text as `find`.
- For `file_patch.find`, copy exact text verbatim from `read_file` or `grep_search` output — never invent or paraphrase.
- CRITICAL: `file_patch.find` must NEVER contain sentinel placeholders, template strings, or invented names (e.g., `__ENTITY_DEFINITION__`, `// function name here`, `<className>`). If you cannot identify a real verbatim anchor from the file content, use `file_append` or `write_file` instead.
- `__PREVIOUS_RESULT__` is ONLY valid in content/value fields — NEVER use it as the `find` anchor in `file_patch`.
- If `grep_search` finds zero matches for your intended anchor, choose a different anchor or use `file_append` instead. Do not retry the same failing patch.
- `file_append` appends content safely before the final `export {}` — use it when a good patch anchor is not obvious.
